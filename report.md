# CST4714 Database Administration Midterm Project
## Campus Event Registration System
## Jahmid Ahmed

This project is a Campus Event Registration System built in PostgreSQL using Supabase. The purpose of the database is to manage students, campus events, and event registrations in a clear and organized way. I chose this topic because it is simple, practical, and easy to understand, but it still allows me to demonstrate important database administration concepts from class. The project focuses on schema design, data integrity, SQL querying, transactions, concurrency, and basic admin/operations tasks.

The schema contains three related tables: `students`, `events`, and `registrations`. The `students` table stores information about each student, including name, email, major, and class level. The `events` table stores the details of each event, such as title, date, location, capacity, and category. The `registrations` table connects students and events. This table represents a many-to-many relationship because one student can register for many events, and one event can have many students. The registrations table uses foreign keys to connect to the primary keys in the students and events tables.

Data integrity was an important part of this project. I used primary keys for all three tables and foreign keys in the registrations table. I also used several constraints to make the database more reliable. For example, student email is marked as `UNIQUE` so duplicate emails cannot be entered. Important fields such as names, email, event title, and location are marked `NOT NULL` so required data cannot be missing. I also added `CHECK` constraints. For example, event capacity must be greater than zero, class level must match one of four allowed values, and registration status must match one of the approved choices. In addition, the registrations table includes a unique constraint on `(student_id, event_id)` so the same student cannot register for the same event more than once.

The project includes sample data to show that the schema works correctly. I inserted fifteen students, fifteen events, and more than fifteen registrations. This provided enough rows for testing joins, aggregates, and views. The queries file contains six SQL queries that show meaningful database use. Two of the queries are joins. These show which students are registered for which events and match event information with student information. Another query uses grouping and counting to show how many registrations each event has. I also used a common table expression to find students who registered for more than one event. A subquery finds events with registrations above the average level. Finally, I created a view called `event_registration_summary` and queried it to show summarized event registration information.

One important part of the project was demonstrating transactions. I used explicit `BEGIN`, `ROLLBACK`, and `COMMIT` statements. In the rollback example, I updated an event capacity and then rolled the transaction back. This showed that the change was not permanently saved. In the commit example, I updated the same event again and committed the transaction. This showed that the second change became permanent. These examples demonstrate the difference between temporary and permanent changes in PostgreSQL transactions.

Another required part of the project was concurrency and blocking. To demonstrate this, I opened two SQL sessions in Supabase. In Session A, I started a transaction and updated one row in the events table without committing. In Session B, I tried to update the same row. Because Session A still held a lock, Session B had to wait. I then ran a diagnostic query using `pg_stat_activity` to view metadata about active sessions. This showed which session was active and which one was waiting. I also used `pg_blocking_pids` as optional evidence to identify the blocking relationship more clearly. After committing Session A, Session B was able to continue. This exercise showed how locks can affect concurrent activity in a database.

For the admin/operations requirement, I created a role called `report_reader` and granted it `SELECT` access to the project tables. I also granted usage on the public schema. Then I verified the granted privileges using `information_schema.role_table_grants`. This step demonstrated a basic access control task that is part of database administration. I also used schema introspection queries from `information_schema.columns` to verify the table structure and column properties.

Overall, this project helped me practice the main topics covered in class so far. The database design is simple, but it demonstrates important concepts clearly. The schema shows relationships and integrity rules, the SQL queries show useful analysis, the transaction section demonstrates commit and rollback behavior, the blocking demo shows concurrency issues and diagnostics, and the role/grant setup shows basic admin control. I learned that even a small database project can show many important database administration skills when the work is organized and well documented.
